1. Introduction to Protocols

What is a Protocol?

A protocol is a set of rules that govern how two or more parties communicate. In security, protocols define the exact sequence of messages exchanged to achieve a security goal like authentication or key exchange.

Types of Protocols

Type Purpose Examples
Communication Protocols Define how data is transmitted HTTP, FTP, SMTP, TCP/IP
Security Protocols Provide security services SSH, SSL/TLS, Kerberos, IPSec

Key Insight: Security protocols must be carefully designed. Even small flaws in protocol design can be exploited by attackers. Unlike bugs in code, protocol flaws are fundamental design errors that are much harder to fix.

2. Simple Security Protocols

Before diving into cryptographic protocols, let's look at everyday authentication examples.

Badge/PIN Authentication

The simplest form of authentication uses something you have (badge) or something you know (PIN).

  • Badge only: Physical access cards — weak if stolen
  • PIN only: Numeric code — weak if observed or guessed
  • Badge + PIN: Two-factor — much stronger (requires both)

ATM Authentication Example

An ATM uses two-factor authentication:

  1. Something you have: The bank card (contains account data on magnetic strip or chip)
  2. Something you know: The PIN (4-6 digit code)
  3. The ATM sends both factors to the bank for verification
  4. If both match, the user is authenticated and can access their account

Vulnerabilities of Simple Protocols

Simple authentication is vulnerable to: shoulder surfing (observing PIN entry), skimming (copying card data), replay attacks (reusing captured authentication data), and social engineering.

3. Authentication with Hash

Hash-Based Authentication

Instead of sending passwords in plaintext, the system stores and verifies a hash of the password. This way, even if the stored data is compromised, the actual passwords are not revealed.

Hash-Based Authentication Flow

Alice
1. "I'm Alice", h(password) →
Server
Alice
← 2. "Welcome" or "Denied"
Server

The Replay Attack Problem

An attacker who captures the hash can simply resend it to authenticate as Alice. The hash never changes!

Solution: Nonces

A nonce (Number used ONCE) is a random value that ensures each authentication attempt is unique and prevents replay attacks.

Hash + Nonce Authentication (Challenge-Response)

Alice
1. "I'm Alice" →
Server
Alice
← 2. Nonce R (random challenge)
Server
Alice
3. h(password || R) →
Server

Why This Works: Because the nonce R is different every time, the hash h(password || R) is different for each session. An attacker who captures one response cannot reuse it.

4. Authentication Using Symmetric Keys

Symmetric Key Authentication

Both parties share a secret key. Authentication is achieved by proving knowledge of the key without revealing it, typically through challenge-response protocols.

Symmetric Key Challenge-Response

Alice
1. "I'm Alice" →
Bob
Alice
← 2. Nonce R
Bob
Alice
3. E(R, KAB) →
Bob

Bob sends a random nonce R. Alice encrypts R with the shared key KAB. Bob decrypts and verifies it matches R.

Why This Works: Only someone who knows the shared key KAB can correctly encrypt the nonce. The nonce prevents replay attacks since it's different each time.

5. Mutual Authentication

What is Mutual Authentication?

Mutual authentication means both parties verify each other's identity. This prevents one side from being fooled by an impersonator.

Mutual Authentication Protocol

Alice
1. "I'm Alice", RA →
Bob
Alice
← 2. RB, E(RA, KAB)
Bob
Alice
3. E(RB, KAB) →
Bob

Alice verifies Bob by decrypting E(RA, KAB) and checking it matches RA. Bob verifies Alice by decrypting E(RB, KAB) and checking it matches RB.

Both Sides Verified: Each party sends a challenge (nonce) and expects the other to encrypt it with the shared key. If both responses are correct, both identities are confirmed.

6. Authentication Using Public Keys

Public Key Authentication

Instead of a shared secret, each party has a public/private key pair. Authentication works by proving possession of the private key (through signing or decryption) without revealing it.

Public Key Authentication Protocol

Alice
1. "I'm Alice" →
Bob
Alice
← 2. R (nonce)
Bob
Alice
3. Sign(R, Alice's Private Key) →
Bob

Bob verifies the signature using Alice's public key. Only Alice's private key could have produced a valid signature on R.

Advantages of Public Key Authentication

  • No shared secret needed: No key distribution problem
  • Digital signatures: Provide non-repudiation (Alice cannot deny signing)
  • Scalable: Each user needs only one key pair

Requires Trusted Public Keys

Public key authentication only works if Bob trusts that Alice's public key truly belongs to Alice. This is solved by certificates issued by a trusted Certificate Authority (CA).

7. Session Keys and Perfect Forward Secrecy

What is a Session Key?

A session key is a temporary symmetric key generated for a single communication session. When the session ends, the key is discarded.

Why Use Session Keys?

  • Limit exposure: If a session key is compromised, only that session is affected
  • Performance: Symmetric keys are fast; use public key only for initial key exchange
  • Fresh keys: Each session gets a unique key, making attacks harder

Perfect Forward Secrecy (PFS)

PFS guarantees that even if a long-term private key is compromised in the future, past session keys cannot be recovered. Each session uses an independent key exchange (e.g., ephemeral Diffie-Hellman).

PFS Example

Without PFS With PFS
Session key derived from server's long-term private key Session key from ephemeral (temporary) key exchange
If private key is leaked later, all past sessions can be decrypted Past sessions remain secure even if private key leaks
RSA key exchange (no PFS) Ephemeral Diffie-Hellman (DHE, ECDHE)

Modern Standard: PFS is now required in TLS 1.3. All key exchanges use ephemeral Diffie-Hellman (ECDHE), ensuring that compromising the server's long-term key does not expose past communications.

Summary and Key Takeaways

What We've Learned

  1. Protocols define the rules of communication. Security protocols must be carefully designed to prevent exploitation.
  2. Simple authentication uses factors like PINs and badges. Combining factors (two-factor authentication) is much stronger.
  3. Hash-based authentication avoids sending passwords in plaintext. Adding a nonce prevents replay attacks.
  4. Symmetric key authentication uses challenge-response: prove you know the shared key by encrypting a random nonce.
  5. Mutual authentication ensures both parties verify each other using nonces and shared keys.
  6. Public key authentication uses digital signatures to prove identity without a shared secret, but requires trusted public key distribution.
  7. Session keys are temporary keys for one session, limiting damage if compromised.
  8. Perfect Forward Secrecy ensures past sessions remain secure even if long-term keys are later compromised.
