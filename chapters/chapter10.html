1. TCP (Transmission Control Protocol)

What is TCP?

TCP is a connection-oriented transport protocol that provides reliable, ordered delivery of data. It underpins most internet applications: HTTP, FTP, SMTP, SSH, and more.

TCP Three-Way Handshake

TCP establishes a connection using a three-step process:

  1. SYN: Client sends a synchronization request
  2. SYN-ACK: Server acknowledges and responds
  3. ACK: Client confirms the connection

TCP Has No Built-in Security

TCP was designed for reliability, not security. It has no encryption, no authentication, and no integrity protection. Attacks include:

  • SYN flood: Denial of service by sending many SYN requests without completing handshakes
  • Session hijacking: Injecting packets into an established TCP connection
  • Eavesdropping: All data transmitted in plaintext

This is why security protocols like SSL/TLS are layered on top of TCP.

2. SSH (Secure Shell)

What is SSH?

SSH provides a secure, encrypted tunnel for remote access. It replaced insecure protocols like Telnet, rlogin, and rsh, which transmitted everything (including passwords) in plaintext.

SSH Features

  • Encrypted communication: All data is encrypted using symmetric ciphers (AES)
  • Server authentication: Client verifies server identity using host keys
  • User authentication: Password, public key, or certificate-based
  • Port forwarding: Tunnel other protocols through the secure SSH connection
  • SFTP/SCP: Secure file transfer built on SSH

SSH in Practice: SSH is the standard tool for server administration. System administrators use SSH keys (public key authentication) instead of passwords for stronger security and automation support.

3. SSL/TLS

What is SSL/TLS?

SSL (Secure Sockets Layer) and its successor TLS (Transport Layer Security) provide encryption, authentication, and integrity for internet communications. TLS is the protocol behind HTTPS.

TLS Handshake (Simplified)

Client
1. ClientHello (supported ciphers, random) →
Server
Client
← 2. ServerHello, Certificate, Key Exchange
Server
Client
3. Key Exchange, ChangeCipherSpec, Finished →
Server
Client
← 4. ChangeCipherSpec, Finished
Server

After the handshake, both parties share session keys and all subsequent data is encrypted.

What TLS Provides

  • Authentication: Server proves identity via digital certificate (signed by CA)
  • Confidentiality: Symmetric encryption (AES) protects data in transit
  • Integrity: MACs detect any tampering
  • PFS: TLS 1.3 requires ephemeral key exchange (ECDHE)

Man-in-the-Middle (MITM) Attacks

Without certificate verification, an attacker can intercept the TLS handshake and present their own certificate. The certificate system (CAs) is designed to prevent this by providing a trusted chain of identity verification.

4. IPSec

What is IPSec?

IPSec (Internet Protocol Security) provides security at the network layer. Unlike SSL/TLS (which secures individual connections), IPSec can protect all traffic between two networks or hosts. It is the foundation of most VPNs.

IPSec Components

Component AH (Authentication Header) ESP (Encapsulating Security Payload)
Authentication Yes Yes
Integrity Yes Yes
Confidentiality No Yes (encrypts payload)
Protects IP header Yes No (in transport mode)

IPSec Modes

  • Transport mode: Protects the payload of the IP packet. Used for end-to-end communication between hosts.
  • Tunnel mode: Protects the entire IP packet (encapsulated in a new packet). Used for VPN gateways connecting networks.

5. Kerberos

What is Kerberos?

Kerberos is a ticket-based authentication protocol. A trusted third party called the Key Distribution Center (KDC) issues tickets that prove a user's identity to services, without transmitting passwords over the network.

Kerberos Authentication Flow

1
Authentication Request

User requests a Ticket Granting Ticket (TGT) from the KDC

↓
2
TGT Issued

KDC verifies identity and issues an encrypted TGT

↓
3
Service Ticket Request

User presents TGT to request access to a specific service

↓
4
Service Ticket Issued

KDC issues a service ticket encrypted for that service

↓
5
Access Granted

User presents service ticket to access the service (mutual authentication possible)

Kerberos in Practice

  • Microsoft Active Directory: Uses Kerberos as its primary authentication protocol
  • Single Sign-On (SSO): Authenticate once, access multiple services via tickets
  • Mutual authentication: Both client and service can verify each other
  • Time-limited tickets: Tickets expire, limiting the window of attack

6. WEP (Wired Equivalent Privacy)

What is WEP?

WEP was the original security protocol for Wi-Fi (IEEE 802.11). It used RC4 stream cipher with CRC-32 for integrity checking.

WEP is Completely Broken

WEP has fatal security flaws:

  • Short IV (24 bits): The Initialization Vector repeats frequently, causing keystream reuse
  • Weak key scheduling: Related-key attacks can recover the WEP key from captured packets
  • CRC-32 is not cryptographic: An attacker can modify packets and fix the CRC without knowing the key
  • Cracking time: A WEP key can be recovered in minutes using freely available tools

Never use WEP. Use WPA2 (AES-CCMP) or WPA3 instead.

7. GSM Security

GSM Authentication

GSM (Global System for Mobile Communications) uses SIM card-based authentication. The SIM stores a unique key (Ki) shared with the network operator.

How GSM Authentication Works

  1. Network sends a random challenge (RAND) to the phone
  2. SIM computes a response using the A3 algorithm: SRES = A3(Ki, RAND)
  3. SIM also computes a session key using the A8 algorithm: Kc = A8(Ki, RAND)
  4. Network verifies SRES; if correct, communication is encrypted using Kc with the A5/1 cipher

GSM Security Weaknesses

  • One-way authentication: The phone authenticates to the network, but the network does not authenticate to the phone (allows fake base stations)
  • Weak encryption: A5/1 has been completely broken
  • No integrity protection: Packets can be modified in transit

Modern standards (3G UMTS, 4G LTE, 5G NR) address these flaws with mutual authentication, stronger encryption (AES), and integrity protection.

Summary and Key Takeaways

What We've Learned

  1. TCP provides reliable data delivery but has no security features. Security must be added by upper layers.
  2. SSH provides encrypted remote access, replacing insecure protocols like Telnet. It uses host keys and supports multiple authentication methods.
  3. SSL/TLS secures web communications (HTTPS) through certificate-based authentication, symmetric encryption, and integrity checking.
  4. IPSec provides network-layer security using AH (authentication) and ESP (encryption + authentication). It is the basis for VPNs.
  5. Kerberos uses ticket-based authentication with a KDC, enabling single sign-on. It is the backbone of Active Directory authentication.
  6. WEP is completely broken due to IV reuse, weak key scheduling, and non-cryptographic integrity. Use WPA2/WPA3 instead.
  7. GSM uses SIM-based challenge-response authentication but suffers from one-way auth and weak encryption. Modern mobile standards fix these issues.
